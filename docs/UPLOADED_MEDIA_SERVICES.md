# Uploaded Media Services

Этот документ описывает service layer для обработки загруженных пользователем медиафайлов: видео, аудио, голосовых сообщений и документов с медиа.

## Область ответственности

Этот слой не занимается:

- маршрутизацией входящих Telegram message;
- маршрутизацией callback payload;
- регистрацией кнопок и callback-id.

Этим занимаются:

- `bot/input_router.py`
- `bot/callback_router.py`
- `bot/handlers.py`
- `bot/callback_registry.py`

Service layer начинается с момента, когда orchestration-слой уже понял, что именно нужно сделать:

- сделать video note;
- сделать voice message;
- расшифровать загруженный файл;
- построить саммари по загруженному файлу.

## Основные модули

- `services/uploaded_media_service.py`
- `services/uploaded_audio_service.py`
- `services/uploaded_video_service.py`
- `services/converter_service.py`

## `uploaded_media_service.py`

Это общий технический фундамент для uploaded media сценариев.

### Что там лежит

- создание временных директорий
- очистка временных директорий
- безопасное обновление status message
- скачивание файла из Telegram по `file_id`
- извлечение аудио из видео
- подготовка аудио из загруженного видео
- подготовка загруженного audio/voice файла

### Почему это вынесено отдельно

И `uploaded_video_service.py`, и `uploaded_audio_service.py` используют одну и ту же базовую механику:

- создать temp dir
- скачать Telegram file
- при необходимости подготовить аудио
- обновлять status message
- убрать временные файлы в `finally`

Если держать это внутри каждого модуля отдельно, код быстро расходится и начинает вести себя по-разному в похожих сценариях.

### Важные инварианты

- `download_telegram_file(...)` требует непустой `telegram_file_id`
- временная директория создаётся заранее или автоматически
- helper'ы не принимают решений уровня продукта, только технически готовят файлы

## `uploaded_audio_service.py`

Отвечает за два сценария:

- `transcribe_uploaded_audio(...)`
- `summarize_uploaded_audio(...)`

### Какие входы реально приходят сюда

- Telegram `audio`
- Telegram `voice`
- audio-документы

С точки зрения сервиса это уже просто Telegram `file_id`. Откуда именно пришёл этот `file_id`, сервису знать не нужно.

### Поток расшифровки

1. проверить наличие `OPENAI_API_KEY`
2. создать temp dir
3. скачать аудиофайл
4. проверить размер против лимита OpenAI
5. отправить файл в OpenAI transcription API
6. разбить длинный текст на чанки
7. отправить пользователю транскрипт
8. отправить `READY_FOR_MORE_TEXT`

### Поток саммари

1. проверить наличие `OPENAI_API_KEY`
2. создать temp dir
3. скачать аудиофайл
4. проверить размер против лимита OpenAI
5. получить транскрипт через OpenAI
6. отдать транскрипт в summary service
7. отправить итоговое саммари
8. отправить `READY_FOR_MORE_TEXT`

## `uploaded_video_service.py`

Отвечает за аналогичные сценарии, но поверх видео:

- `transcribe_uploaded_video(...)`
- `summarize_uploaded_video(...)`

### Отличие от audio service

Перед OpenAI здесь есть дополнительный технический шаг:

- скачать видео
- убедиться, что в нём есть аудиодорожка
- извлечь аудио через `ffmpeg`

Поэтому для video service нужен `ffmpeg`, а для audio service — нет.

### Почему video и audio сервисы разделены

Хотя в них есть похожие участки, разделение здесь полезно:

- user-facing тексты разные
- precheck'и разные
- failure modes разные
- поведение “в видео нет аудио” — отдельный сценарий

То есть это не бессмысленное дублирование, а два похожих, но разных продукта.

## `converter_service.py`

Это модуль преобразований формата, а не транскрипции.

### Что он делает

- `convert_video_file_to_video_note(...)`
- `convert_audio_file_to_voice_message(...)`

### Video note

Поток:

1. проверить `ffmpeg`
2. скачать видео из Telegram
3. определить параметры видео через `ffprobe`
4. при необходимости обрезать до 59 секунд
5. привести к квадратному crop
6. последовательно попробовать encoding profiles
7. отправить результат как Telegram video note

### Voice message

Поток:

1. проверить `ffmpeg`
2. скачать аудиофайл из Telegram
3. перекодировать его в `ogg/opus`
4. проверить размер
5. отправить как Telegram voice message

### Что важно

Сейчас оба конвертера возвращают `True/False`, а не просто “молча пытаются”. Это важно для orchestration-слоя: `handlers.py` не должен редактировать status message как успешный, если конвертация реально не получилась.

## Ограничения и лимиты

### Лимит OpenAI на транскрипцию

Используется `OPENAI_TRANSCRIPTION_FILE_LIMIT`, сейчас это около `25 MB`.

Это ограничение влияет на:

- `uploaded_audio_service.py`
- `uploaded_video_service.py` после извлечения аудио

### Лимит Telegram на отправку файлов

Используется `MAX_FILE_SIZE`, сейчас это около `50 MB`.

Это ограничение влияет на:

- отправку audio из YouTube/Instagram
- отправку voice message
- другие outbound media сценарии

### Ограничение video note

Для video note дополнительно есть внутренний “безопасный” лимит размера файла, чтобы Telegram стабильно принимал результат. Поэтому конвертер умеет пробовать более компактные encoding profiles.

## Ошибки и честное поведение

Важный принцип для этого слоя: сервисы должны отвечать пользователю честно и предсказуемо.

Примеры:

- если нет `OPENAI_API_KEY`, сервис сразу говорит об этом
- если нет `ffmpeg`, сервис не делает вид, что проблема “в файле”
- если в видео нет аудио, пользователю говорится именно это
- если аудио больше лимита OpenAI, пользователю показывается причина
- если OpenAI вернул пустой результат, это не маскируется как успех

## Границы между слоями

### Что должен делать handler

- понять, какой сценарий нужен
- показать кнопки
- достать или сохранить `file_id`
- запустить background task

### Что должен делать service

- скачать файл
- подготовить данные
- поговорить с OpenAI / `ffmpeg`
- отправить пользователю итог

### Что не должен делать service

- решать, какая кнопка была нажата
- разбирать `callback_data`
- распознавать YouTube/Instagram URL

## Почему текущая структура считается хорошей

На текущем объёме проекта сервисная часть выглядит удачно по нескольким причинам:

- shared low-level helper'ы уже вынесены
- orchestration и execution разделены
- uploaded audio/video сценарии изолированы друг от друга
- форматные конверсии лежат отдельно от OpenAI сценариев
- сервисы покрыты unit-тестами

При этом код ещё не ушёл в слишком абстрактную generic-framework архитектуру, где читаемость часто страдает сильнее, чем выигрывает повторное использование.

## Что здесь сознательно не делалось

### Не вводилась единая super-generic media pipeline abstraction

Это можно сделать, но на текущем размере проекта она бы, скорее всего, начала скрывать реальные различия между:

- video transcription
- video summary
- audio transcription
- audio summary
- video note conversion
- voice message conversion

То есть общности здесь есть, но они пока лучше выражаются helper-модулями, а не глубокой иерархией generic workflow classes.

### Не убиралось всё дублирование до нуля

Некоторая доля повторения в service layer сейчас оправдана, потому что:

- тексты разные
- precheck'и разные
- пользовательские ошибки разные
- критерии отказа разные

## Как безопасно расширять эту часть дальше

### Если появится новый uploaded media сценарий

Например:

- OCR по картинке
- вырезание звука из видео в отдельный файл
- нормализация аудио

Лучший путь:

1. оставить routing в `bot/`
2. в `services/` добавить отдельный сценарий
3. shared file/status helper'ы переиспользовать из `uploaded_media_service.py`

### Если появится новый вид конверсии

Например:

- audio -> mp3
- video -> gif

Лучше класть это в `converter_service.py` или в соседний conversion-oriented модуль, а не смешивать с transcription/summary сервисами.
