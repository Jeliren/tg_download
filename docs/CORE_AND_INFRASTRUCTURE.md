# Core And Infrastructure

Этот документ описывает базовый инфраструктурный слой проекта: конфигурацию, startup/shutdown runtime, concurrency-примитивы и общие low-level helper'ы.

## Зачем выделять это отдельно

Эти модули не делают user-facing продуктовые сценарии сами по себе, но от них зависит поведение всего бота:

- как стартует процесс;
- как он переживает ошибки polling;
- как останавливается по сигналу;
- как ограничиваются фоновые задачи;
- как чистятся временные файлы;
- как логируются события и скрываются секреты.

Если этот слой хаотичный, проблемы начинают проявляться уже во всех остальных частях проекта.

## Основные модули

- `config.py`
- `main.py`
- `core/cache.py`
- `core/task_registry.py`

- `utils/logging_utils.py`

## `config.py`

`config.py` остаётся простым module-level settings-файлом без лишнего framework-слоя.

### Что он делает

- подхватывает `.env` из корня проекта;
- не перетирает уже заданные переменные окружения;
- приводит bool/int значения к ожидаемым типам;
- хранит runtime defaults;
- даёт базовую валидацию через `validate_config()`;
- возвращает non-fatal предупреждения через `get_runtime_warnings()`.

### Что важно

- `.env` читается один раз при импорте модуля;
- shell-style строки вида `export KEY=value` тоже поддерживаются;
- `validate_config()` проверяет только обязательный минимум;
- отсутствие `OPENAI_API_KEY` не считается ошибкой старта, потому что OpenAI-сценарии в проекте optional.

### Polling-related настройки

Слой запуска использует отдельные настройки:

- `POLLING_TIMEOUT`
- `LONG_POLLING_TIMEOUT`
- `MAX_POLLING_RESTARTS`
- `POLLING_RESTART_DELAY`

Они вынесены в config, чтобы в production не держать критичное runtime-поведение в магических числах внутри `main.py`.

## `main.py`

`main.py` отвечает только за жизненный цикл процесса, а не за Telegram-бизнес-логику.

### Текущая последовательность старта

1. создать runtime-директории;
2. настроить логирование;
3. зарегистрировать signal handlers;
4. очистить старые временные файлы;
5. вывести runtime warnings;
6. создать и настроить bot instance;
7. залогировать `event=runtime_started`;
8. войти в polling loop с ограниченным числом restart attempts.

### Shutdown model

Для остановки используется `ShutdownController`.

Его задача специально узкая:

- пометить, что shutdown запрошен;
- при наличии bot instance вызвать `stop_polling()`;
- не делать тяжёлую cleanup-логику прямо внутри signal handler.

Это важно, потому что cleanup и `sys.exit(...)` прямо из signal handler делают runtime менее предсказуемым и хуже тестируются.

### Почему текущий подход удачный

- нет import-side effects вроде регистрации signal handlers при простом `import main`;
- cleanup temp-файлов сосредоточен в `finally`, а не размазан по разным веткам;
- `runtime_started` логируется после успешной инициализации бота;
- polling restart loop ограничен и настраивается из config.

## `core/cache.py`

`ExpiringStore` — это маленький потокобезопасный TTL-кеш в памяти.

### Где он уместен

- callback registry;
- короткоживущие runtime-кеши;
- временные технические данные, которые не должны переживать рестарт процесса.

### Важные инварианты

- кеш защищён `RLock`;
- TTL считается по `time.monotonic()`, а не по wall clock;
- отрицательный TTL считается ошибкой конфигурации и не допускается.

Использование monotonic clock важно, потому что обычный `time.time()` может “прыгать” при сдвиге системного времени.

## `core/task_registry.py`

`UserTaskRegistry` — самый простой lock на пользователя.

Он не пытается быть очередью задач, scheduler'ом или полноценной state machine. Для текущего проекта это правильный уровень сложности:

- один пользователь не должен запускать несколько тяжёлых операций одновременно;
- глобальная orchestration-логика остаётся в `handlers.py`;
- сам реестр остаётся тривиальным и надёжным.

## `core/task_runner.py`

`BackgroundTaskRunner` — ограниченный thread pool для фоновых задач.

### Что он гарантирует

- количество одновременных задач ограничено;
- очередь не растёт бесконечно;
- при переполнении новые задачи честно отклоняются;
- после `shutdown()` новые задачи не принимаются;
- ошибки внутри worker'ов логируются и не валят вызывающий поток.

### Что здесь сознательно не делалось

- не добавлялась отдельная job framework abstraction;
- не вводилась сложная приоритизация задач;
- не делался persistent queue layer.

Для Telegram-бота этого масштаба bounded `ThreadPoolExecutor` даёт хороший компромисс между надёжностью и простотой.

## `utils/logging_utils.py`

Здесь лежит общий runtime logging.

### Что важно

- все сообщения проходят через sanitizer;
- bot token редактируется в логах;
- есть структурированный `log_event(...)` в формате `key=value`;
- perf-замеры используют `time.perf_counter()`, а не wall clock;
- внешние noisy-библиотеки приглушаются до `WARNING`.

### Практический смысл

Проекту не нужен отдельный observability stack, но нужны:

- читаемые логи;
- маскирование секретов;
- возможность быстро понять, какой сценарий упал и на каком шаге.

Текущий слой это обеспечивает без лишней сложности.

## `utils/file_utils.py`

Этот модуль объединяет общие технические helper'ы вокруг Telegram IO и временных файлов.

### Что там лежит

- `send_with_retry(...)`
- progress-message helpers
- `cleanup_temp_folder(...)`

### Что важно про retry

`send_with_retry(...)` нужен как тонкий защитный слой над Telegram send-операциями:

- повторяет только ограниченное число раз;
- не повторяет заведомо бесполезный сценарий `Too big file`;
- не делает лишнюю паузу после последней неудачной попытки.

### Что важно про temp cleanup

`cleanup_temp_folder(...)`:

- умеет работать с явным `temp_dir`;
- удаляет содержимое, а не саму директорию;
- отказывается чистить заведомо опасные пути вроде `/` и домашней директории.

Это не “защита от всего”, но полезный предохранитель от особенно неприятных ошибок.

## Архитектурные границы

### Что относится к infrastructure

- process lifecycle
- logging
- temp files
- bounded background execution
- short-lived caches
- config parsing

### Что не относится к infrastructure

- parsing user intent;
- callback routing;
- YouTube/Instagram бизнес-логика;
- OpenAI pipelines;
- медиа-конвертация как продуктовый сценарий.

Это важно держать раздельно, чтобы инфраструктурный слой не превращался в “god module”.

## Как развивать дальше

Если меняется infrastructure, сначала стоит проверить один вопрос:

“Эта сложность действительно общая для всего проекта, или это локальная потребность одного сценария?”

Хорошие причины править этот слой:

- новый повторяющийся runtime-risk;
- проблема graceful shutdown;
- неконтролируемый рост очередей или temp-файлов;
- необходимость единообразного structured logging;
- реальный дефект в retry/cleanup logic.

Плохие причины:

- “давайте завернём всё в ещё один abstraction layer”;
- “давайте введём DI-контейнер, потому что так моднее”;
- “давайте сделаем generic runtime framework на будущее”.

Для текущего проекта infrastructure уже должен быть аккуратным, но оставаться небольшим и понятным.
